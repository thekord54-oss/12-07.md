### Задание 1

Опишите основные преимущества использования масштабирования методами:

    активный master-сервер и пассивный репликационный slave-сервер;
• Максимальная отказоустойчивость (High Availability): Главная цель пассивного слейва — подстраховка. Он работает в режиме «горячего резерва» (Hot Standby), непрерывно и в фоновом режиме копируя бинарные логи с активного мастера.
• Быстрое аварийное переключение (Failover): Если основной сервер (Master) выходит из строя из-за аппаратного сбоя, пассивный слейв мгновенно переводится в статус Master. Приложение перенаправляет трафик на него, благодаря чему система продолжает работу без потери данных.
• Безопасное создание бэкапов: Тяжелые операции резервного копирования можно запускать на пассивном сервере, полностью исключая просадки производительности на основном рабочем сервере.

    master-сервер и несколько slave-серверов;
• Масштабирование производительности чтения: Все ресурсоемкие операции чтения (SELECT) распределяются между несколькими ведомыми серверами. Это позволяет эффективно обрабатывать лавинообразный рост пользовательских запросов.
• Изоляция нагрузки (Load Balancing): Операции изменения данных (INSERT, UPDATE, DELETE) выполняются исключительно на мастере, не мешая пользователям быстро читать данные со слейвов.
• Оптимально для Read-Heavy систем: Эта архитектура идеальна для веб-ресурсов, где пользователи преимущественно просматривают информацию (каталоги, профили, статьи), а пишут данные редко. Выход из строя одной реплики никак не нарушает общую доступность данных.

### Задание 2

Для примера возьмем три таблицы со следующими полями:

    пользователи: id_user (номер), name (имя), email.

    книги: id_book (номер), title (название), author (автор), price (цена).

    магазины: id_store (номер), store_name (название магазина), address (адрес).

1. Вертикальный шардинг (Разделение по таблицам)
Принцип работы: Мы просто берем разные таблицы и разносим их по отдельным физическим серверам на основе их логики.

В нашем случае делим так:

    Сервер А (База данных "Пользователи"): Хранит только таблицу пользователи.
    Сервер Б (База данных "Каталог книг"): Хранит только таблицу книги.
    Сервер В (База данных "Магазины"): Хранит только таблицу магазины.

    Зачем это нужно: Чтобы запросы не мешали друг другу. Например, когда сотни людей одновременно ищут книги в каталоге, Сервер Б будет загружен, но это никак не затормозит Сервер А, и новые пользователи смогут спокойно регистрироваться.


2. Горизонтальный шардинг (Разделение строк внутри таблицы)

Принцип работы: Когда одна конкретная таблица становится слишком гигантской (например, пользователей стало 50 миллионов), её уже нельзя хранить на одном сервере. Мы берем строки этой таблицы и режем их на части.

В качестве ключа деления (ключа шардирования) возьмем id_user. Разделим таблицу пользователи на два сервера (шарда) по простому принципу — четный или нечетный ID:

    Шард 1 (Сервер Пользователи-1): Сюда записываются пользователи, у которых id_user нечетный (1, 3, 5, 7...).
    Шард 2 (Сервер Пользователи-2): Сюда записываются пользователи, у которых id_user четный (2, 4, 6, 8...).

Приложение само понимает, на какой сервер стучаться: если заходит юзер с ID 15, запрос идет на первый сервер, если с ID 20 — на второй.

3. В каких режимах работают сервера

Чтобы система была надежной и не теряла данные, каждый наш рабочий сервер мы подстрахуем репликацией:

    Режим Master (Основной): Все наши главные серверы шардов (и для книг, и для магазинов, и оба шарда для пользователей) работают как Master. Приложение может и писать туда новые данные, и читать их.

    Режим Slave (Резервный / Только для чтения): К каждому Мастеру мы подключаем по одному запасному серверу в режиме Slave (с настройкой read_only = 1). Они в фоновом режиме копируют всё за своим мастером. Если какой-то Мастер сломается, его Slave сразу займет его место, и данные не потеряются.
